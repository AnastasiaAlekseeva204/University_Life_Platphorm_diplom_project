# FAQ: Типовые вопросы и ответы на защиту ВКР

## Вопросы по базам данных

### Q: Почему вы использовали SQLite в начале разработки?
**A:** SQLite был использован на ранних этапах как встроенная БД для быстрого прототипирования и локального тестирования без необходимости запуска отдельного сервера БД. SQL Lite подходит для проектирования схемы, разработки моделей и тестирования логики. Однако для production система была перенесена на PostgreSQL благодаря его масштабируемости и производительности при работе с большим количеством пользователей.

### Q: Какие отличия SQLite от PostgreSQL?
**A:**
| Параметр | SQLite | PostgreSQL |
|----------|--------|-----------|
| **Архитектура** | Встроенная БД (хранится в одном файле) | Клиент-серверная архитектура |
| **Масштабируемость** | Ограничена, подходит до 1-2 GB данных | Высокая, поддерживает петабайты данных |
| **Конкурентность** | Слабая поддержка одновременных запросов | Отличная поддержка параллельных операций |
| **ACID транзакции** | Базовая поддержка | Полная ACID поддержка |
| **Пользователи и права** | Нет встроенной системы ролей | Развитая система пользователей и прав доступа |
| **Индексы** | B-tree | B-tree, BRIN, Hash, GiST, GIN |
| **Реплика и резервные копии** | Нет встроенной поддержки | Встроенная функция replication |
| **JSON поддержка** | Базовая | Полная (jsonb, json операторы) |

Вывод: SQLite идеален для малых и средних приложений, PostgreSQL — для enterprise решений и высоконагруженных систем.

## Вопросы по безопасности веб-приложений

### Q: Что такое CSRF (Cross-Site Request Forgery) и как его предотвратить?
**A:** **CSRF** — атака, при которой злоумышленник заставляет пользователя выполнить действие на его имя без его одобрения.

**Пример:** Если пользователь авторизован на сайте A.com, а затем случайно откроет вредоносный сайт B.com, последний может отправить запрос на A.com от его имени (например, перевод денег).

**Защита в Django:**
- Django по умолчанию генерирует CSRF-токен для каждого сеанса
- Токен должен отправляться в POST, PUT, DELETE запросах
- Промежуточное ПО (middleware) `CsrfViewMiddleware` проверяет токен
- Пример HTML: `<form method="post">{% csrf_token %}</form>`

### Q: Что такое XSS (Cross-Site Scripting) и как его предотвратить?
**A:** **XSS** — атака, при которой злоумышленник внедряет вредоносный JavaScript код на страницу, который выполняется в браузере пользователя.

**Пример:** Если сайт выводит нефильтрованный контент пользователя: `<p>{{ user_input }}</p>`, и пользователь введет `<script>alert('XSS')</script>`, скрипт выполнится.

**Защита:**
- **Escaping (экранирование)**: Django по умолчанию экранирует переменные в шаблонах: `{{ variable }}` → безопасно
- **Использование фильтров**: `{{ content|escape }}`
- **CSP (Content Security Policy)**: Заголовок HTTP, который ограничивает источники скриптов
- **Валидация входных данных**: Проверияте и очищайте пользовательский ввод на backend
- **Пример уязвимого кода:**
  ```django
  {{ user_comment|safe }}  {# ОПАСНО! Отключает экранирование #}
  ```
- **Безопасный код:**
  ```django
  {{ user_comment }}  {# Безопасно, автоматически экранируется #}
  ```

## Вопросы по парсингу веб-данных

### Q: Как работает парсер в вашем приложении?
**A:** Парсер извлекает данные о мероприятиях с сайта МПГУ (https://mpgu.su/anonsyi/) и сохраняет их в БД.

**Этапы работы:**
1. **Загрузка страницы**: `requests.get(url)` — получаем HTML
2. **Парсинг DOM**: `BeautifulSoup(html, 'html.parser')` — преобразуем HTML в дерево объектов
3. **Извлечение данных**: Используем CSS-селекторы (`soup.select()`) для поиска нужных элементов:
   - Заголовок: `soup.select_one('h1.deep-blue')`
   - Дату: `soup.select_one('h3')` и парсим формат
   - Изображение: `soup.find('img')`
   - Текст: `soup.select_one('.qwen-markdown-text')`
4. **Обработка данных**: Очищаем от тегов, форматируем дату
5. **Сохранение**: Создаем объект `ParsedEvent` в БД
6. **Обработка ошибок**: Try-except для сетевых сбоев и ошибок парсинга

**Особенности:**
- Проверяем уникальность по `source_url` (unique=True)
- Извлекаем первое подходящее изображение, исключая логотипы
- Ограничиваем размер текста (max_length)
- Используем timeout=15 сек для HTTP-запросов

### Q: Какие вызовы при парсинге и как вы их решили?
**A:**
- **Динамический контент**: Некоторые данные загружаются JavaScript. Решение: Использовали статические HTML-селекторы, т.к. основной контент статичен.
- **Изменяющаяся структура HTML**: Сайт может менять структуру. Решение: Использовали регулярные выражения и множественные селекторы.
- **Производительность**: Парсинг всех событий занимает время. Решение: Асинхронная обработка через Django команды.
- **Ошибки сети**: Сайт может быть недоступен. Решение: Try-except, логирование ошибок, повторные попытки.

## Вопросы по архитектуре приложения

### Q: Почему вы выбрали Django для проекта?
**A:**
- **Встроенная админка**: Управление данными без дополнительных инструментов
- **ORM**: Удобная работа с БД через Python объекты
- **Безопасность**: Встроенная защита от CSRF, XSS, SQL-injection
- **Масштабируемость**: Матрица приложений позволяет легко расширять функционал
- **Документация**: Отличная документация и большое сообщество
- **Rapid development**: Быстрая разработка с минимумом кода

### Q: Как спроектирована модель данных?
**A:** Используется реляционная модель с normalization (до 3НФ):
- **User**: Пользователь с ролью (ADMIN, STUDENT, ORGANIZER)
- **Event**: Мероприятие c организатором (FK) и участниками (M2M)
- **Community**: Сообщество аналогично Event
- **Faculty**: Иерархическая структура (MPPT для вложенности)
- **ParsedEvent**: Спарсенные события с связью на User через participants

**Связи:**
- Один-ко-многим: Event → User (organizator)
- Многие-ко-многим: Event ↔ User (participants), ParsedEvent ↔ User
- Иерархия: Faculty → Faculty (parent)

Все модели поддерживают created_at/updated_at для аудита.

## Вопросы по функциональности

### Q: Как работает рейтинговая система?
**A:** Каждый пользователь имеет `rating_points` в профиле. Баллы начисляются за:
- Участие в событии: +5 баллов
- Участие в сообществе: +3 балла
- Волонтёрские часы: +10 за 1 час (admin устанавливает вручную)

Рейтинг пересчитывается при каждом участии. Система мотивирует студентов быть активнее.

### Q: Как реализована авторизация?
**A:** Django использует встроенную систему аутентификации:
- Пароли хешируются (PBKDF2)
- Сессии хранятся в БД или Redis
- Ролевой контроль доступа: @login_required, limit_choices_to={'role': User.Role.ORGANIZER}
- JWT токены не используются, но можно добавить для API

### Q: Как парсер интегрирован с админкой?
**A:** 
- Django команда: `python manage.py parse_mpgu_events` запускает парсер вручную
- Можно добавить Celery для периодического запуска (cron-style)
- Админка позволяет редактировать спарсенные события через CKEditor
- Можно публиковать события в Telegram через ботов

## Дополнительные ответы для подготовки

### Q: Как вы защищены от SQL-injection?
**A:** Django ORM использует prepared statements:
```python
# Безопасно
user = User.objects.filter(username=request.GET['user'])
# Опасно (не используйте!)
users = User.objects.raw(f"SELECT * FROM user WHERE username='{request.GET['user']}'")
```

### Q: Что такое middleware в Django?
**A:** Промежуточное ПО, которое обрабатывает запросы/ответы. В проекте используются:
- CsrfViewMiddleware (CSRF защита)
- SessionMiddleware (сессии)
- AuthenticationMiddleware (авторизация)

### Q: Как работает ORM Django?
**A:** Object-Relational Mapping преобразует SQL в Python код:
```python
# Вместо: SELECT * FROM event WHERE date > '2026-04-15'
events = Event.objects.filter(date_time__gt=timezone.now())
```
