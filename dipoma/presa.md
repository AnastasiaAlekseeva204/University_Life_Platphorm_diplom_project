## Блок 1. Проблема и целевая аудитория (`present.md` п. 1)

### Слайд 2. Какую проблему решает проект

- **Проблема:** информация о внеучебных мероприятиях разбросана (соцсети, сайты, объявления); нет единого места для записи, учёта участия и мотивации через рейтинг/портфолио.
- **Зачем платформа:** снижение потерь информации, прозрачность для администрации, удобство для студентов.

### Слайд 3. Целевая аудитория

- **Основная:** студенты МПГУ; организаторы мероприятий; администрация (деканаты, студсоветы).
- **Дополнительная:** сотрудники, волонтёры; возможность масштабирования на другие вузы.
- **Потребности:** актуальные анонсы, регистрация, учёт участия, рейтинги, единый интерфейс.

*Устно:* «для кого» и «какую боль» закрываем — одним абзацем.

## Блок 2. Состояние вопроса, конкуренты, новизна (`present.md` п. 2)

### Слайд 4. Состояние проблемы

- Распространённые подходы: фрагментированные каналы (VK, Moodle-объявления, бумажные списки).
- **Ограничения:** нет связки «официальный источник → автоматический сбор → единый каталог → учёт и мотивация» в одной системе.

### Слайд 5. Решения конкурентов / аналоги

- **Moodle / LMS:** учебный контент, не специализация на внеучебной активности и портфолио достижений.
- **Внешние соцсети:** охват есть, но нет институционального учёта и ролевой модели «студент / организатор / админ».
- **Зарубежные кампус-платформы:** социальные функции; слабее учёт достижений и интеграция с локальными источниками вуза.

### Слайд 6. Новизна вашего проекта

- **Автоматизация:** парсинг анонсов с официального ресурса (BeautifulSoup, requests) → модель анонсов в БД.
- **Персонализация:** роли, профиль, факультет, участие в событиях и сообществах.
- **Мотивация и учёт:** рейтинговые механизмы, связь участия с данными профиля (логика портфолио/достижений — по вашей реализации).
- **Интеграция компонентов:** единый Django-проект (приложения `myplatform`, `events_parser`).

*Устно:* чётко: «чем мы отличаемся от Moodle + группы ВК».

## Блок 4. Дизайн проекта: структура, модули, цель, эффекты (`present.md` п. 4)

### Слайд 9. Цель проекта и ожидаемые эффекты

- **Цель:** централизованная платформа для анонсов, регистрации и учёта внеучебной активности.
- **Количественные эффекты (пример — подставьте свои цифры из пилота):** сокращение времени поиска мероприятий; доля пользователей, нашедших событие за N минут; время отклика страниц; число обработанных анонсов.
- **Качественные эффекты:** рост вовлечённости, прозрачность для администрации, формирование цифрового следа активности.

### Слайд 10. Структура проекта (архитектура)

- **Backend:** Django (`myplatform`, `events_parser`).
- **Frontend:** шаблоны, Bootstrap.
- **БД:** SQLite (разработка) / PostgreSQL (целевой прод).
- **Медиа:** изображения профилей, событий, сообществ.
- *(Схема: клиент → Django → ORM → БД; парсер → модели анонсов.)*

### Слайд 11. Примеры модулей / фрагментов

- **Модели:** `User` (роли), `Event`, `Community`, `Faculty`, `ParsedEvent` — кратко поля и связи.
- **Парсер:** загрузка страницы, извлечение даты, описания, HTML, URL изображения; сохранение в `ParsedEvent`.
- **UI:** `event_detail.html`, списки событий/сообществ, профиль.
- **Админка:** управление сущностями.

*Устно:* 1 скрин или фрагмент кода — «доказательство реализации».
